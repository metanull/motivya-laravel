# MVP Payment Ledger Recovery

## Epic

Title: Make successful Stripe payments visible across bookings, finance, transfers, and payout statements

Context: A production Bancontact test payment on 2026-05-13 succeeded in Stripe for booking #3 and Checkout Session `cs_test_a1W02duDzWJzVy4YuGQ6atvWRp1BSIrxe6c18pzDolM6XwaWEFtgVGvfRv`. Stripe reports PaymentIntent `pi_3TWjNBCZkAo8qMcM0x17b5Ca`, amount 1500 EUR cents, status `succeeded`, Checkout Session status `complete`, and payment status `paid`. The app confirmed the booking and recorded financial audit events, but `bookings.stripe_payment_intent_id` remained null, no invoice or coach payout statement exists, the accountant financial dashboard stays at zero because it is invoice-based, and the coach cannot see an app-level transfer or payment statement. Production also contains booking #2 with the same missing-payment-intent shape.

Problem: The application currently allows successful Stripe payments to become only partially represented in Motivya. The Stripe webhook endpoint is not subscribed to `checkout.session.completed`, even though `StripeWebhookController` implements a handler for it. The fallback `payment_intent.succeeded` handler confirms bookings by metadata but does not persist the PaymentIntent ID. The `payments:reconcile-bookings --repair` command retrieves Checkout Sessions with `payment_intent` expanded, then accepts only string payment-intent values, so it incorrectly reports that recoverable sessions have no payment intent. Stripe creates transfer events for destination charges, but the production webhook endpoint is not subscribed to `transfer.created` and the app has no transfer ledger. Payout statements are based only on completed sessions and are never generated by a scheduled command, while the accountant dashboard summary cards are invoice-based and do not explain collected payments that are not yet invoiced.

Solution: Normalize the Stripe payment lifecycle into one reliable ledger path. Successful Checkout and PaymentIntent webhooks must always persist the PaymentIntent ID and amount. Reconciliation must repair recoverable existing bookings whether Stripe returns a string or expanded PaymentIntent object. Stripe webhook endpoint configuration must be verified in readiness checks and include all events handled by the app. Destination-charge transfer events must be stored and surfaced to coach/accountant users. Payout statements must be generated deterministically for completed-session periods. Dashboards must distinguish collected payment, invoice, transfer, anomaly, and payout-statement states instead of collapsing everything into invoice totals.

Cross-cutting constraints:

- All monetary values remain integer EUR cents.
- Stripe keys and webhook secrets must only be read through `config()` and never printed in logs, UI, stories, tests, or commands.
- Booking confirmation and reconciliation must be idempotent and safe under repeated Stripe event delivery.
- Payment, transfer, invoice, and payout statement changes must record financial audit events.
- User-facing strings must be localized in `lang/fr`, `lang/en`, and `lang/nl`.
- No provider failure may be silently ignored; readiness/admin tools must expose actionable Stripe configuration errors.

Child stories:

- Story 1: Persist PaymentIntent IDs from all successful Stripe payment webhooks.
- Story 2: Repair the booking reconciliation command and backfill affected production bookings safely.
- Story 3: Enforce Stripe webhook subscription readiness for all handled finance events.
- Story 4: Record Stripe transfers and expose them in coach/accountant finance views.
- Story 5: Generate monthly coach payout statements from completed paid sessions.
- Story 6: Split accountant dashboard metrics into collected payment, invoice, anomaly, transfer, and payout-statement states.

## Story 1

Title: Persist PaymentIntent IDs from successful Stripe payment webhooks

Context: `App\Http\Controllers\StripeWebhookController` handles both `checkout.session.completed` and `payment_intent.succeeded`. Production processed `payment_intent.succeeded` for booking #3 as event `evt_3TWjNBCZkAo8qMcM0BUoeiC7`, confirmed the booking, and set `amount_paid = 1500`, but left `stripe_payment_intent_id = null`. The Checkout Session completion event existed in Stripe as `evt_1TWjNMCZkAo8qMcMoTlQaK6J`, but the production webhook endpoint was not subscribed to it.

Problem: The PaymentIntent success path can confirm a paid booking without storing the PaymentIntent ID. Confirmed paid bookings missing `stripe_payment_intent_id` become anomalous, cannot be refunded through Stripe, cannot be reconciled by downstream finance workflows, and make admin/accountant/coach views disagree.

Solution: Make the webhook controller persist the PaymentIntent ID and amount whenever Stripe reports a successful payment, regardless of whether the app receives `checkout.session.completed` or `payment_intent.succeeded` first.

Implementation Instructions:

1. Update `app/Http/Controllers/StripeWebhookController.php` so `handlePaymentIntentSucceeded()` stores `stripe_payment_intent_id` from `$paymentIntent->id` when confirming a pending booking.
2. When `payment_intent.succeeded` finds a booking by metadata rather than an existing `stripe_payment_intent_id`, persist the PaymentIntent ID before dispatching `BookingCreated`.
3. Keep the handler idempotent: if the booking is already confirmed and has the same PaymentIntent ID, do nothing except allow the processed webhook row to remain recorded.
4. If the booking is already confirmed, has `amount_paid > 0`, and is missing `stripe_payment_intent_id`, allow the handler to fill the missing ID when metadata matches the booking.
5. Update `handleCheckoutSessionCompleted()` so it accepts a string PaymentIntent ID and an expanded Stripe PaymentIntent object. Store the object `id` when the webhook payload is expanded.
6. Record `AuditEventType::BookingPaymentConfirmed` for initial confirmation and `AuditEventType::BookingPaymentReconciled` when an already confirmed booking is repaired by webhook replay or delayed delivery.
7. Add or update Pest tests covering `payment_intent.succeeded` with metadata, `checkout.session.completed` with string `payment_intent`, `checkout.session.completed` with expanded `payment_intent`, duplicate webhook delivery, and delayed repair of an already confirmed booking missing the PaymentIntent ID.

Definition of Done:

- Successful `payment_intent.succeeded` webhooks persist `stripe_payment_intent_id` and `amount_paid` on the matching booking.
- Successful `checkout.session.completed` webhooks persist `stripe_payment_intent_id` whether Stripe sends a string or expanded object.
- Replayed successful webhooks are idempotent and do not duplicate audit records beyond the existing processed-webhook guard.
- Confirmed paid bookings missing a PaymentIntent ID can be repaired by a later matching successful webhook.
- Relevant automated tests are added or updated for the business logic and user-visible behavior.
- Lint checks pass without warnings or errors.
- Relevant test suites pass without warnings or errors.

## Story 2

Title: Repair booking reconciliation for expanded Stripe PaymentIntents

Context: The production command `php artisan payments:reconcile-bookings --repair` found bookings #2 and #3 as confirmed paid bookings missing PaymentIntent IDs. Stripe retrieval for booking #3 returns Checkout Session `paid/complete` with expanded `Stripe\PaymentIntent` object `pi_3TWjNBCZkAo8qMcM0x17b5Ca`, but `App\Console\Commands\ReconcileBookings` checks only `is_string($stripeSession->payment_intent)`, so it incorrectly reports `checkout session has no payment intent`.

Problem: The repair command cannot repair the exact recoverable state it creates by requesting `expand => ['payment_intent']`. Operators receive a false manual-review result even when Stripe has a succeeded PaymentIntent with matching metadata.

Solution: Normalize the retrieved Checkout Session payment-intent value before repair and backfill affected confirmed paid bookings only when Stripe confirms a single safe match.

Implementation Instructions:

1. Update `app/Console/Commands/ReconcileBookings.php` to extract a PaymentIntent ID from either a non-empty string or an expanded Stripe object with a non-empty `id`.
2. Validate that the retrieved Checkout Session is `status = complete`, `payment_status = paid`, and `amount_total` equals `bookings.amount_paid` before repair.
3. Validate that Checkout Session metadata `session_id` and `athlete_id` match the booking before repair.
4. Keep the existing collision guard that prevents assigning a PaymentIntent already used by another booking.
5. In `--repair` mode, update `stripe_payment_intent_id` and record `AuditEventType::BookingPaymentReconciled` inside a transaction.
6. Add command output that distinguishes `not paid`, `amount mismatch`, `metadata mismatch`, `collision`, `stripe retrieval failed`, and `repaired`.
7. Add a focused test that reproduces the production shape: expanded PaymentIntent object returned from Stripe and a confirmed paid booking with null `stripe_payment_intent_id`.
8. After deployment, run the command in dry-run mode, then repair mode, and verify bookings #2 and #3 no longer appear in the unreconciled list.

Definition of Done:

- The command repairs Checkout Sessions whose `payment_intent` is an expanded Stripe object.
- The command refuses repairs when payment status, amount, metadata, or collision checks fail.
- Repaired bookings receive a financial audit event.
- Production bookings #2 and #3 can be safely backfilled after deployment if Stripe data still matches.

## Story 3

Title: Verify Stripe webhook subscriptions in readiness checks

Context: Production has one enabled Stripe webhook endpoint at `motivya.metanull.eu/stripe/webhook`. Its enabled events include `payment_intent.created`, `payment_intent.succeeded`, invoice events, and `account.updated`, but do not include `checkout.session.completed`, `payment_intent.payment_failed`, `charge.refunded`, or `transfer.created`. The application code handles several of these missing events.

Problem: The app can implement a Stripe handler that production never receives. This causes partial state: Checkout completion logic is bypassed, refunds may not generate credit-note side effects, failed payments may not release holds, and transfers are invisible to the app.

Solution: Add a Stripe webhook readiness check that compares the code-required finance event list against the configured Stripe endpoint and reports missing subscriptions as an actionable admin readiness error.

Implementation Instructions:

1. Define a single code-owned list of required Stripe webhook event types for the MVP finance flow: `checkout.session.completed`, `payment_intent.succeeded`, `payment_intent.payment_failed`, `charge.refunded`, `account.updated`, and `transfer.created`.
2. Add a Stripe webhook endpoint inspection service under `app/Services/Stripe/` or the existing Stripe service namespace. It must use the Stripe SDK and `config('services.stripe.secret')` only.
3. Update `app/Livewire/Admin/Readiness.php` to include a Stripe webhook subscription check that reports green when all required events are enabled on the active endpoint and red when any required event is missing.
4. Add localized readiness messages in all supported languages. The message must name the missing event types and instruct the operator to update the Stripe webhook endpoint.
5. Do not auto-modify Stripe webhook endpoints from the app. This story is read-only readiness and operator guidance only.
6. Add tests for all-required-events-present, missing-events, no-enabled-endpoint, and Stripe API failure.
7. Update production runbook documentation with the exact required Stripe event list.

Definition of Done:

- Admin readiness reports missing Stripe webhook event subscriptions before new payments are accepted as healthy.

Production session #10 is still `published` for 2026-05-27, so no statement is expected yet, but completed sessions also will not receive statements automatically without an explicit generator.

7. Update coach payout-statement UI copy so coaches understand that statements are generated after sessions are completed and the month is closed.

## Story 6

Title: Separate collected payment metrics from invoice dashboard metrics

Context: `App\Livewire\Accountant\Dashboard` summary cards are based on `Invoice` rows. A newly paid booking for a future or not-yet-completed session has no invoice, so revenue, VAT, payout pending, and invoice counts remain zero. The accountant transaction ledger can show booking-level payment activity, while the dashboard summary cards only reflect invoice-generation activity.

Problem: A successful payment can be visible in transactions and audit logs while the dashboard remains all zeros. This is technically consistent with invoice-based metrics, but misleading during operations because collected cash, missing invoices, missing payout statements, and anomalies are not separated.

Solution: Redesign accountant finance summary metrics to show distinct states: collected successful payments, invoice revenue, pending payout statements, Stripe transfer records, open anomalies, and sessions awaiting invoice/statement generation.

Implementation Instructions:

1. Update `app/Livewire/Accountant/Dashboard.php` to add booking-based current-month collected payment metrics using confirmed bookings with `amount_paid > 0`.
2. Keep existing invoice-based metrics, but label them as invoice metrics in localized UI strings.
3. Add metrics for confirmed paid bookings missing PaymentIntent IDs, paid completed sessions without invoices, completed paid sessions without payout statements for the target period, and paid bookings without recorded Stripe transfer rows after Story 4 is implemented.
4. Update `resources/views/livewire/accountant/dashboard.blade.php` to group summary cards by collected payments, invoices, payouts, transfers, and anomalies.
5. Ensure the date filters apply consistently or explicitly label metrics that always use the current calendar month.
6. Add tests proving a future confirmed paid booking increments collected payment totals but not invoice totals.
7. Add tests proving a completed paid session without an invoice appears in the waiting-for-invoice metric.
8. Add tests proving missing PaymentIntent and missing transfer states are visible without altering invoice totals.

Definition of Done:

- The accountant dashboard no longer appears empty when successful payments exist but invoices do not yet exist.
